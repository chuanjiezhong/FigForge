# FigForge 阶段A实现方案调研

## 一、需求概述

**阶段A目标**：本地单机版 - 桌面端 + 本地 R + 排版器

### 核心功能模块

1. **计算引擎（Analysis Engine）**
   - 调用用户本机 R / Rscript
   - 支持多 pipeline（单细胞、bulk、显微图等）
   - 支持 pipeline 版本化
   - JSON 参数输入
   - 进度回调与日志输出
   - 统一输出目录结构（result.json, panels/*.svg, tables/*.csv, artifacts.json）

2. **渲染引擎（Rendering Engine）**
   - 阶段一：R/ggplot 输出 SVG
   - 确保分析结果的可信性与科研接受度

3. **排版编辑器（Layout Editor）**
   - 支持 SVG/PNG/TIFF 等多种 panel 类型
   - 支持多 panel 画布（A/B/C…）
   - 拖拽、缩放、裁剪、旋转
   - 自动对齐、吸附、等间距
   - 自动 panel 标号
   - 文本、箭头、比例尺等标注
   - 全局字体、字号、线宽控制
   - 保存与恢复排版状态（layout.json）
   - 所有排版操作不触发重新计算

4. **导出引擎（Export Engine）**
   - 支持 SVG / PDF 矢量导出
   - 支持 PNG / TIFF 位图导出
   - 支持 300 dpi / 600 dpi
   - 支持期刊模板（Nature / 单双栏）
   - 支持字体嵌入与颜色规范
   - 支持一键导出投稿版本

## 二、技术栈选型调研

### 2.1 桌面应用框架选择

#### 方案A：Electron
**优点：**
- 跨平台（Windows/macOS/Linux）
- 成熟生态，大量成功案例（VS Code, Slack, Discord等）
- 可以使用 Web 技术栈（React/Vue/Angular）
- 丰富的 npm 包生态
- 易于开发和调试

**缺点：**
- 体积较大（~100MB+）
- 内存占用相对较高
- 性能不如原生应用

**推荐度：⭐⭐⭐⭐⭐**

#### 方案B：Tauri
**优点：**
- 体积小（~10MB）
- 性能好（使用系统 WebView）
- 内存占用低
- Rust 后端，安全性高
- 跨平台支持

**缺点：**
- 相对较新，生态不如 Electron 成熟
- 需要 Rust 知识（后端）
- 文档和社区资源较少

**推荐度：⭐⭐⭐⭐**

#### 方案C：PyQt/PySide
**优点：**
- Python 生态，易于集成 R 调用
- 原生性能
- 成熟稳定

**缺点：**
- 界面开发相对复杂
- 跨平台需要额外配置
- 现代化 UI 需要更多工作

**推荐度：⭐⭐⭐**

#### 方案D：Flutter Desktop
**优点：**
- 现代化 UI 框架
- 跨平台
- 性能好

**缺点：**
- 桌面端相对较新
- 需要 Dart 语言
- 与 R 集成需要额外工作

**推荐度：⭐⭐⭐**

**推荐方案：Electron + React/Vue**
- 开发效率高
- 生态成熟
- 易于实现复杂的排版编辑器
- 社区支持好

### 2.2 前端框架选择

#### 方案A：React + TypeScript
**优点：**
- 生态成熟，组件库丰富
- TypeScript 类型安全
- 适合复杂交互应用
- 大量 UI 库（Material-UI, Ant Design等）

**推荐度：⭐⭐⭐⭐⭐**

#### 方案B：Vue 3 + TypeScript
**优点：**
- 学习曲线平缓
- 性能优秀
- 组合式 API 适合复杂逻辑

**推荐度：⭐⭐⭐⭐**

**推荐方案：React + TypeScript**
- 更适合复杂的数据流和状态管理
- 排版编辑器需要复杂的状态管理

### 2.3 R 调用方案

#### 方案A：Node.js child_process
**优点：**
- 原生支持，无需额外依赖
- 可以实时获取输出和错误
- 支持进度监控

**实现方式：**
```javascript
const { spawn } = require('child_process');
const rProcess = spawn('Rscript', ['script.R', '--params', jsonParams]);
```

**推荐度：⭐⭐⭐⭐⭐**

#### 方案B：r-script (npm包)
**优点：**
- 封装好的 R 调用库
- 简化使用

**缺点：**
- 依赖第三方库
- 可能不够灵活

**推荐度：⭐⭐⭐**

#### 方案C：Python 桥接（通过 Python 调用 R）
**优点：**
- rpy2 等库成熟
- Python 生态丰富

**缺点：**
- 增加一层依赖
- 复杂度提升

**推荐度：⭐⭐**

**推荐方案：Node.js child_process**
- 直接、高效
- 完全控制 R 进程
- 易于实现进度回调和日志捕获

## 四、关键技术实现要点

### 4.1 R 调用实现

**核心需求：**
- 调用本地 Rscript
- JSON 参数传递
- 进度回调
- 日志捕获
- 统一输出结构

**实现思路：**
1. 通过 child_process.spawn 启动 Rscript
2. 参数通过命令行或临时 JSON 文件传递
3. R 脚本通过 stdout/stderr 输出进度和日志
4. 解析 R 输出，提取进度信息
5. 监控输出目录，等待结果文件生成

### 4.2 排版编辑器实现

**核心需求：**
- 多 panel 画布
- 拖拽、缩放、旋转
- 自动对齐、吸附
- 标注工具
- 状态保存

**实现思路：**
1. 使用 Canvas（Konva/Fabric）或 SVG 作为画布
2. 每个 panel 作为独立对象
3. 实现拖拽、缩放、旋转的交互逻辑
4. 实现对齐线和吸附功能
5. 标注作为独立图层
6. 所有操作序列化为 layout.json

### 4.3 导出引擎实现

**核心需求：**
- 多格式导出（SVG/PDF/PNG/TIFF）
- 高 DPI 支持
- 期刊模板
- 字体嵌入

**实现思路：**
1. SVG：直接序列化画布内容
2. PDF：使用 Puppeteer 渲染 HTML/SVG 到 PDF
3. PNG/TIFF：使用 Sharp 转换 SVG 或 Canvas
4. 模板：预定义尺寸和布局配置
5. 字体：确保系统字体可用，或嵌入字体文件

## 六、潜在挑战与解决方案

### 挑战1：R 环境检测与配置
**问题：** 用户可能没有安装 R 或路径配置不正确
**解决：** 
- 启动时检测 R 环境
- 提供 R 安装指引
- 允许用户手动配置 R 路径

### 挑战2：R 脚本执行时间较长
**问题：** 长时间运行可能阻塞 UI
**解决：**
- 使用 Web Worker 或独立进程
- 显示进度条和日志
- 支持取消操作

### 挑战3：大文件处理性能
**问题：** 高分辨率图片可能导致内存问题
**解决：**
- 使用流式处理
- 图片压缩和缓存
- 懒加载机制

### 挑战4：跨平台兼容性
**问题：** Windows/macOS/Linux 差异
**解决：**
- 使用 Electron 的跨平台 API
- 测试各平台 R 调用
- 处理路径差异
